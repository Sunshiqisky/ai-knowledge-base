我做 fuxi 的第一天，干了件特别蠢的事。

我打开 Claude Code，敲了一句：「帮我做一个个人知识库，能抓链接、自动整理、还能搜索。」回车，转身去倒了杯咖啡。

说句实话，那次失败教会我关于 Vibe Coding 最重要的一件事：**问题不在 AI，在我不会指挥它。**

## 01 Vibe Coding 不是魔法，是你当包工头

先把这个词讲清楚。

**Vibe Coding** 是 AI 大神卡帕西 2025 年初造的词，字面意思是「跟着感觉走地写代码」——你别管代码细节，把活儿交给 AI，你只盯着「做出来的东西对不对」。

听起来很爽。于是很多人理解歪了，以为 Vibe Coding 是：**我说一句话，AI 给我吐一个 App。**

不是。我第一天就是这么想的，结果你也看到了。

**类比：你是包工头，AI 是施工队。** 你不用自己砌墙、布线、刷漆（写代码），但你得干三件事：说清楚要盖成啥样（讲需求）、一层一层验收（做验证）、哪面墙歪了让工人返工（调 bug）。你要是甩一句「给我盖栋楼」就走人，回来保证是一堆废墟。

我那天就是个甩手包工头。

| | ❌ 甩手掌柜（我的第一天） | ✅ 称职包工头（你该做的） |
|---|---|---|
| 怎么开口 | 一句话「做个知识库」 | 先喂图纸，再分步派活 |
| 进度 | 让它一口气写完 | 一次只做一小块 |
| 验收 | 全做完才跑，一跑全崩 | 每做一步跑一步 |
| 出错时 | 一堆错，不知从哪查 | 错少、好定位、能返工 |

记住这张表，因为接下来整篇，我都在教你怎么从左边那列，变成右边那列。具体就是**四步**：

![Vibe Coding 四步循环](../_assets/10-vibe-coding-loop@2x.png)

（这张图是这篇的骨架：喂上下文 → 分步推进 → 每步验证 → 卡住止损，是一个不断转的圈，不是一条直线——每做完一小步，就转回来再派下一步。）

这里有个让你松一口气的真相：**整个过程，你不用看懂任何一行代码。**

你是包工头，不是工人。你要会的是「说清楚、会验收、能返工」，不是「会砌墙」。代码长啥样，交给 AI；东西跑没跑通，你说了算。

> 💡 **一句话总结**：Vibe Coding 不是「说句话变出 App」，是你当包工头——说清需求、分步验收、出错返工，代码细节一行都不用懂。

## 02 开工第一步：把那三份文档喂给 AI

Vibe Coding 的成败，八成在你敲第一个字之前就定了。

定在哪？**上下文**——也就是「你有没有先把图纸给施工队」。没图纸的施工队，全凭想象，盖出来的肯定不是你要的。我第一天那句「做个知识库」，就是让 AI 凭空想象，它当然给我一坨四不像。

好消息是，你前面三篇的功课，现在全派上用场了：

- **07 产品思维** → 你产出过一份「一页纸 MVP」，写清楚做给谁、解决啥
- **08 架构思维** → 你产出过一份架构文档，写清楚「东西怎么走」
- **09 技术思维** → 你产出过一份选型清单，写清楚用啥工具

**这三份，就是你交给施工队的图纸。**

怎么喂？很简单——建一个空项目文件夹，把这三份文档丢进去，然后开口第一句不是派活，是让它先读：

```
我要做一个个人知识库。这个文件夹里有三份文档：
产品定义、架构、技术选型。
请先把它们读完，用你自己的话复述一遍我要做的是什么，
确认我们理解一致后，再等我派第一个任务，先别写代码。
```

这一句里藏了两个关键动作：**让它复述**（检验它真读懂了没）、**让它先别写代码**（你还没派活，别让它擅自开干）。

我现在每开一个新项目，第一件事都是这个——先喂文档、让它复述、对齐认知。就这一步，能帮你省掉后面 80% 的「它怎么做的跟我想的完全不一样」。

> 💡 **一句话总结**：开口第一句不是「给我写」，是「先读我的图纸、复述一遍对齐」——上下文喂到位，后面才不跑偏。

## 03 第一个高光：扔个链接，AI 当场给你生成一篇笔记

对齐完，咱们做 fuxi 的第一个、也是最核心的功能：**ingest**——你扔个链接进来，它自动抓取、读懂、整理成一篇笔记。

这里我要先告诉你一件，可能又一次颠覆你认知的事：

**这个功能，一行代码都没写。**

你没看错。fuxi 里「抓链接 → 生成笔记」这套，不是一个程序，而是**一段写给 AI 看的规则**。你把规则讲清楚，AI 每次照着做就行。规则写在一个叫 `CLAUDE.md` 的文件里（还记得 04、05 讲的 Skill 和工作流吗？就是它的延续）。

规则长这样，你复制下来改改就能用：

```
收到一个链接时，按这个流程处理：

1. 打开链接，把正文从头读完
2. 判断它在讲什么，提炼成 2-3 句话的摘要 + 3-5 个要点
3. 给它打 3-5 个标签（不够就少打，别硬凑）
4. 存成一个 Markdown 文件，放进 notes/ 文件夹
   文件名用「日期-标题.md」
   文件开头写好：标题、原链接、标签、日期
```

就这么几行大白话。然后你真的扔一个链接进去，AI 当场就给你吐出一篇笔记：

```markdown
---
title: 某篇讲 AI Agent 的文章
url: https://...
tags: [AI, Agent, 工作流]
date: 2026-06-11
---

## 摘要
这篇文章讲了 Agent 的三个核心能力……（2-3 句）

## 要点
- 要点一
- 要点二
- 要点三
```

成了。你没写代码，AI 就把「读文章、提炼、打标签、存文件」这一整套全干了。

这就是 Vibe Coding 里最反直觉、也最爽的一点：**有些活根本不用写程序，你把要求说清楚，AI 当场就替你干了。**

当然，我也在这儿栽过。第一版规则我写得太死，要求「每篇必须打 5 个标签」。结果有些文章明明俩标签就讲完了，AI 为了凑够 5 个，硬编出一堆不相干的词。后来我改成「3-5 个，不够就少打」，立马正常了。

**规则是你给 AI 的需求说明书：写太死，它就僵；写太松，它就飘。** 这个度，得你来把。

> 💡 **一句话总结**：fuxi 最核心的「抓链接生成笔记」零代码——一段大白话规则交给 AI 照做就行；规则别写太死，留点余地它才不僵。

## 04 该写代码了：让 AI 写一个「能搜」的脚本

笔记攒到几十篇，你就想搜了：「我之前存的那篇讲 Agent 的在哪？」

这时候你当然也能直接问 AI「帮我搜 Agent」。但它每次都得把所有文件从头扫一遍，**又慢又费 token**。这种「规则很明确、要反复干」的机械活，最适合落成一个脚本——写一次，以后一秒出结果。

这一节，咱们真要写代码了。但说好的——**你还是不用自己写，也不用看懂，你只管指挥。**

指挥的核心纪律就一条：**别让它一口气写完，拆成小步，一步一跑。**

别这么说：「写个搜索功能。」（这就是我第一天的死法）

要这么说，一步一步来：

```
第一步：先写一个脚本，能列出 notes/ 里所有笔记的标题。先做这个，做完我跑一下。
```

跑通了，再派下一步：

```
第二步：现在加一个功能，我给一个关键词，只列出标题里包含这个词的笔记。
```

再跑通，再下一步：

```
第三步：现在让它不只搜标题，也搜每篇的标签和摘要。
```

每加一步，你就在终端跑一下，对了再往下。最后你就得到一个能用的搜索：

```
$ python3 search.py Agent

找到 3 条：
  2026-06-11-某篇讲AI-Agent的文章.md
  2026-05-28-Agent工作流设计.md
  2026-05-10-多Agent协作.md
```

整个过程，我一行 Python 都没给你看——因为你不需要看。你只需要知道：**你说了第几步，它跑出了什么。** 这就是包工头验收，不是工人砌墙。

这里再送你一个我用真金白银换来的教训。第一版搜索，我贪心，一句话让 AI「做个搜索，支持关键词、标签、日期范围、正则、模糊匹配」。它真给我全写了，参数一大堆。结果呢？我日常 99% 的时候就用关键词那一个，其余全是摆设。后来我砍到只剩「关键词 + 标签」两个，清爽多了。

**贪多，是 Vibe Coding 的第一杀手。** 先做最小的那个能用版本，缺啥再加，永远比一上来求大求全强。这正是 09 反复砸的那句——**优先走轻量方案。**

> 💡 **一句话总结**：机械活才写脚本，写也得拆小步、一步一跑、代码不用看懂；先做最小能用版，别一上来求大求全。

## 05 报错了怎么办：这才是 Vibe Coding 最真实的样子

我得跟你说句大实话：**只要你动手，就一定会报错。** 没有例外。

报错本身不可怕，可怕的是你一看到满屏红字就慌、就觉得「我果然不行」。打住——你是包工头，墙砌歪了是工人返工，不是你的锅。

而你要做的，就一个动作：**把报错原样复制，整段甩回给 AI，说「报这个错，帮我修」。**

不用看懂报错，一个字都不用懂。

给你看段我自己的真实实录。当时写完搜索，我一搜中文关键词，啪，蹦出一串：

```
UnicodeDecodeError: 'utf-8' codec can't decode byte ...
```

我压根不知道这是啥。我直接复制，甩回给 Claude：「报这个错，帮我修。」它说是文件编码的问题，改一行就好。我一跑——又报另一个错。再甩回去。就这么来回 4 轮，通了。前后没超过十分钟，我全程没看懂任何一个错。

调 bug 你只要守住三条纪律：

| | ❌ 新手（越调越乱） | ✅ 老手（越调越顺） |
|---|---|---|
| 怎么报错 | 自己用话转述「它好像打不开文件」 | 报错**原样复制**甩回去，一字不改 |
| 改几处 | 让它一次改五个地方 | 一次只改一处，改完立刻跑 |
| 卡住时 | 同一个错死磕半天，越改越乱 | 来回 3 轮还不好，就让它「这段重写」 |

第三条我得多说一句。有一次一个 bug，我和 Claude 缠了快一个钟头，改了又改，越改越乱，文件都快不成样子了。最后我一咬牙：「别修了，把这个文件整个删了，按原来的需求重写一遍。」5 分钟，搞定。

**记住：有时候，重来比修快。** 别因为「已经改了这么多」就舍不得推倒——沉没成本不值钱，跑通才值钱。

> 💡 **一句话总结**：报错原样甩回去、一次只改一处、卡住超过 3 轮就让它重写——你全程不用看懂任何一个错。

## 06 一条贯穿全篇的规律：语义活交给 AI，机械活才写脚本

回头看你今天做的两件事，藏着 fuxi 的全部秘密：

- **ingest（抓链接生成笔记）**——没写代码，一段规则交给 AI 当场干
- **search（搜索）**——写了脚本，落成一个一秒出结果的程序

凭啥一个写代码、一个不写？区别就一条：

| | 语义活（交给 AI 当场干） | 机械活（落成脚本） |
|---|---|---|
| 特点 | 要理解、要判断 | 规则明确、反复重复 |
| 例子 | 读懂文章、提炼观点、打标签、判断哪两篇有关联 | 扫文件、关键词匹配、生成索引、统计标签 |
| 怎么实现 | 写进 `CLAUDE.md` 的大白话规则 | 让 AI 写一个脚本 |
| 为啥这么分 | 这些活只有「会理解」的 AI 干得了 | 这些活机械、高频，脚本快又稳、还省 token |

这就是 09 结尾那句「机械归脚本、语义归 Claude」落到地上的样子。

往大了说，这其实揭示了 AI 时代做产品最反常识的一点：**你的大部分「开发」，根本不是在写代码，而是在写清楚需求。**

你今天写得最多的，不是 Python，是那段大白话规则。这跟 04 讲的 Skill、05 讲的工作流，是同一条线——**在 AI 时代，能把「要什么」说清楚，比会写代码值钱得多。**

而「把要什么说清楚」这件事，恰恰不需要你是程序员。它需要的是产品思维、是想清楚（06-09 那四篇干的事）。看到了吗？绕了一大圈，全都接上了。

> 💡 **一句话总结**：语义活（理解、判断）交给 AI 写进规则，机械活（重复、明确）才落成脚本——AI 时代的「开发」，大半是把需求说清楚，不是敲代码。

## 07 动手环节：今天就 vibe 出你的第一个小功能

光看不练，三天就忘。今天就动手，做一个你自己的、最小的能跑的东西。

**前提**：你得有 Claude Code 或 Codex 其中一个（没装的话，文末有上手指引，十分钟搞定）。

然后三步：

1. **建个空文件夹**，把你 07、08、09 那三份文档丢进去，让 AI 先读完、复述一遍对齐
2. **挑一个最小的功能**——别挑大的，就挑「扔个链接、生成一篇笔记」这种，写 5 行大白话规则让 AI 干
3. **扔一个真实输入进去**，看它跑出啥；报错就原样复制甩回去，来回几轮治通

就这么简单。记住唯一的纪律：**选最小的那个，先跑通一个**。跑通一个最小的，比憋一个大的强一百倍——因为跑通的那一下，你会真正相信「原来我也能做出东西」，这个信心，比任何教程都重要。

> 💡 **一句话总结**：建空文件夹 → 喂三份文档对齐 → 挑最小功能写 5 行规则 → 扔真实输入治通，今天就跑出你的第一个东西。
