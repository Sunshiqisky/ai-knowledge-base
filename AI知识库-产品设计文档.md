## 〇、产品定位

### 一句话定位

> 我做的是一个给 **写作者/内容创作者** 用的 **AI 原生个人知识库**，跟 **Notion/Obsidian/收藏夹** 不一样的地方是 **AI 自动整理 + 往回织更新 + 主动推荐选题**。

### AI Native 判断（✅ 全部满足）

| 标准 | 判断 |
|------|------|
| 核心价值由 AI 提供 | ✅ 摘要、打标签、关联推荐、选题生成全靠 AI |
| 交互以自然语言为主 | ✅ 直接扔链接/想法，AI 当场处理 |
| 越用越懂你 | ✅ 往回织机制，每次新增资料都会更新相关主题 |

### 需求来源（三把铲子挖出来的）

| 铲子 | 具体痛点 |
|------|----------|
| 每天打开 5 次以上的 App | Notion、Obsidian、微信收藏——存了一堆但找不到 |
| 重复劳动 | 每周手动整理素材、手动分类、手动写摘要 |
| 骂娘时刻 | 选题时翻半小时找不到之前存的资料 |

## 一、项目定义（MVP）

| 维度 | 内容 |
|------|------|
| **用户** | 我自己 |
| **问题** | 素材散落、选题时找不到、想发散没人帮、AI 完全没参与 |
| **输入** | 我手动丢进去的链接、想法 |
| **处理** | Claude 读完 → 写带立场的摘要 → 更新索引 → 反向更新相关条目 |
| **输出** | 一个能问、能查、能告诉我「今天可以写哪篇」的本地知识库 |
| **不做什么** | 不做数据库、不做配图、不做协作、不做多端、不做音视频识别 |

### 砍功能三把刀（从 30+ 砍到 5 个）

| 刀 | 砍掉的功能 | 理由 |
|----|-----------|------|
| **第一刀：不是核心** | 协作分享、多端同步 | 自用不需要，团队功能 |
| **第二刀：现在能不要** | 周报、自动配图、定时提醒 | 一个月后加也来得及 |
| **第三刀：不做不会死** | 微信对接、语音输入、灵感发散对话 | 锦上添花，后续逐步对接 |

## 二、架构设计（进、存、取）

### 2.1 进：收什么、怎么进来

| 维度 | 答案 |
|------|------|
| **来源** | 链接（GitHub / X / 公众号）、PDF、想法 |
| **触发** | 我手动丢 |
| **格式** | URL / PDF 文件 / 自然语言 |

### 2.2 存：分层组织（四层结构）

| 层级 | 目录 | 存什么 | 说明 |
|------|------|--------|------|
| **原料层** | `raw/` | 抓回来的原文、PDF | 原样存着、只读不改——事实来源，AI 判断错了能翻回来核对 |
| **笔记层** | `notes/` | AI 写的带判断的读书笔记 + 我的想法笔记 | 每条素材一篇笔记，想法单独存 |
| **编译层** | `wiki/` | 主题页 + 实体页 | **灵魂层**：多篇聊同一件事的笔记合并成一页「结论」 |
| **选题层** | `briefs/` | 选题清单 | 把结论变成「这周能写哪篇」的可执行清单 |

**核心机制：往回织**
- 每加一条新资料，AI 会回头把相关的主题页、实体页都重写一遍
- 会标出新旧资料的矛盾
- 不是简单加个链接

### 2.3 取：三条输出流

| 流 | 触发 | 输出形式 |
|----|------|----------|
| **问答流** | 我问什么 | AI 先查目录 → 找到相关「结论页」→ 直接读那页 → 综合回答 |
| **选题流** | 基于主题页 | 告诉我「这个主题现在能写哪几篇」，直接给可执行选题 |
| **周报流** | 每周定时 | 梳理「本周入库了什么」+「哪个主题该编译了」 |

### 2.4 完整数据流

```
链接 / PDF ──→ raw/ 原文 ──→ notes/ 单源笔记（带立场）
                                    ↓
想法 / 灵感 ──→ notes/ 想想笔记 ──→ 编译层（主题页 / 实体页）
                                      ↑↓ 往回织（标矛盾）
                                      ↓
                              ┌───────┼───────┐
                              ↓       ↓       ↓
                           问答流   选题流   周报流
```

## 三、技术选型

### 3.1 选型四原则

| 原则 | 说明 |
|------|------|
| **主流** | 选 AI 训练数据多的语言（Python / TypeScript） |
| **本地** | 能不上云就别上，零服务器、零部署、零运维 |
| **简单** | 文件 > 数据库，能不用就不用 |
| **大众** | star 多 + 更新近 + 维护者活 |

### 3.2 具体技术栈

#### 抓取层（按渠道挑，没有银弹）

| 渠道 | 工具 | 说明 |
|------|------|------|
| 普通网页 / 博客 | Claude Code 内置 WebFetch | 零额外依赖 |
| 微信公众号 | Jina Reader | 托管 reader API，抓下来直接是干净 Markdown |
| GitHub | `gh CLI` | 官方命令行，登录后直接读 README、issue、release |
| YouTube | `yt-dlp` | 视频抓取库 star 最多（80k+） |
| X / Twitter | `opencli twitter` 系列 | 社区现成工具 |

#### 存储层

| 项目 | 选型 | 说明 |
|------|------|------|
| 主数据 | 纯 Markdown 文件 | 人能读、git 能 diff、AI 能直接喂 |
| 目录分层 | `raw/` + `notes/` + `wiki/` + `briefs/` | 按「生熟程度」分层 |
| 数据库 | ❌ 没用 | 文件 + Markdown frontmatter 完全够 |
| 向量库 / embedding | ❌ 没用 | 个人知识库完全不需要 |

#### 索引 + 编排层

| 项目 | 选型 | 说明 |
|------|------|------|
| 自动索引 | Python 小脚本（AI 写） | 扫所有笔记的 frontmatter，自动生成 INDEX.md |
| 关联推荐 | Python 小脚本（AI 写） | 「标签重叠 + 标题关键词」打分推荐，双向写入 `related` 字段 |
| 检索 | 关键词搜索脚本 + Claude 长上下文 | 关键词够准，语义判断扔给 Claude |
| AI 模型 | Claude（命令行调用） | 写中文笔记最准 |
| 工作流编排 | Claude 语义工作流（写在 `CLAUDE.md`） | 不写复杂状态机，让 AI 直接读规则执行 |
| 运行环境 | 本地 Mac | 零服务器、零部署、零运维 |

### 3.3 不上向量也能找到关联

**核心机制：AI 打标签 → 脚本算重叠 → 双向写入文件**

1. **AI 打标签**：Claude 在 ingest 每篇笔记时自动打 3-5 个标签（不是手打，是语义判断）
2. **脚本算重叠**：拿新笔记跟所有老笔记两两比较，算「标签重叠数 + 标题关键词重叠数」，加权打分
3. **双向写入**：分数 ≥ 4 的推荐为关联，双向写入 frontmatter 的 `related` 字段

```yaml
---
title: Karpathy LLM Wiki 理念
related:
  - 2026-05-10-cyrilXBT-JARVIS知识系统.md
  - 2026-05-08-Garry-Tan-GBrain操作系统.md
---
```

**为什么比向量更好？**
- 标签已经是语义压缩过的
- 标题是你自己最浓缩的判断
- 可解释——分数怎么算的清清楚楚

## 四、实施步骤（Vibe Coding 四步法）

### 4.1 核心纪律

| 步骤 | 说明 |
|------|------|
| **喂上下文** | 把三份文档（MVP、架构、选型）喂给 AI，让它复述一遍对齐 |
| **分步推进** | 一次只做一小块，别让它一口气写完 |
| **每步验证** | 每做一步跑一步，对了再往下 |
| **卡住止损** | 报错原样甩回去、一次只改一处、卡住超过 3 轮就让它重写 |

### 4.2 分工铁律

| 类型 | 特点 | 实现方式 |
|------|------|----------|
| **语义活** | 要理解、要判断 | 写进 `CLAUDE.md` 的大白话规则，交给 AI 当场干 |
| **机械活** | 规则明确、反复重复 | 让 AI 写一个脚本，落成程序 |

### 4.3 第一步：Ingest（抓链接生成笔记）

**零代码实现**，写进 `CLAUDE.md` 的规则：

```
收到一个链接时，按这个流程处理：

1. 打开链接，把正文从头读完
2. 判断它在讲什么，提炼成 2-3 句话的摘要 + 3-5 个要点
3. 给它打 3-5 个标签（不够就少打，别硬凑）
4. 存成一个 Markdown 文件，放进 notes/ 文件夹
   文件名用「日期-标题.md」
   文件开头写好：标题、原链接、标签、日期
```

**输出格式示例：**

```markdown
---
title: 某篇讲 AI Agent 的文章
url: https://...
tags: [AI, Agent, 工作流]
date: 2026-06-11
---

## 摘要
这篇文章讲了 Agent 的三个核心能力……（2-3 句）

## 要点
- 要点一
- 要点二
- 要点三
```

### 4.4 第二步：Search（搜索）

**分步实现：**

```
第一步：先写一个脚本，能列出 notes/ 里所有笔记的标题。先做这个，做完我跑一下。

第二步：现在加一个功能，我给一个关键词，只列出标题里包含这个词的笔记。

第三步：现在让它不只搜标题，也搜每篇的标签和摘要。
```

**输出示例：**

```
$ python3 search.py Agent

找到 3 条：
  2026-06-11-某篇讲AI-Agent的文章.md
  2026-05-28-Agent工作流设计.md
  2026-05-10-多Agent协作.md
```

### 4.5 调 Bug 纪律

| 场景 | 正确做法 |
|------|----------|
| 报错 | 原样复制甩回去，一字不改 |
| 改几处 | 一次只改一处，改完立刻跑 |
| 卡住时 | 来回 3 轮还不好，就让它「这段重写」 |
| 越改越乱 | 整个文件删了，按原来的需求重写一遍 |

**记住：有时候，重来比修快。**

## 五、项目结构

```
my-knowledge-base/
├── CLAUDE.md              # AI 工作流规则（语义活）
├── ARCHITECTURE.md        # 架构文档（本文档）
├── TECH-STACK.md          # 选型清单
├── INDEX.md               # 自动生成的索引
├── raw/                   # 原料层：原文、PDF（只读）
├── notes/                 # 笔记层：AI 写的带判断的笔记
├── wiki/                  # 编译层：主题页、实体页
├── briefs/                # 选题层：选题清单
└── scripts/               # 脚本：索引、关联、搜索
    ├── generate_index.py
    ├── find_related.py
    └── search.py
```

## 七、核心心法

### 产品思维两大心法

| 心法 | 说明 |
|------|------|
| **找需求** | 去你天天用的产品里找金矿，只做痛点不碰痒点 |
| **做形态** | 小众赛道 + AI Native，大厂顾不上的就是你的机会 |

### 四阶段心法

| 阶段 | 心法 |
|------|------|
| 产品思维 | MVP 是滑板车不是半辆车，「不做什么」比「做什么」更重要 |
| 架构思维 | 先画「取」再画「进」，只画东西不画工具 |
| 技术思维 | 优先走轻量方案，能不用就不用 |
| Vibe Coding | 语义活交给 AI，机械活才写脚本 |

**一句话总结：先把烂版本跑起来，再去想优化。**
